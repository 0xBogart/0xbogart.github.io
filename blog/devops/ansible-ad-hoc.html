∅ ¿POR QUÉ AD HOC?

Un comando ad hoc de Ansible utiliza la herramienta de línea de comandos /usr/bin/ansible para automatizar una única tarea en uno o más nodos gestionados.

Los comandos ad hoc son rápidos y fáciles, pero no son reutilizables.

Entonces, ¿por qué aprender primero los comandos ad hoc? Los comandos ad hoc demuestran la simplicidad y el poder de Ansible. Los conceptos que se aprenden aquí se trasladan directamente al lenguaje de playbook.

Son ideales para tareas que se repiten con poca frecuencia. Por ejemplo, si quieres apagar todas las máquinas de un laboratorio para las vacaciones, puedes ejecutar una línea rápida en Ansible sin escribir un playbook. Un comando ad hoc tiene este aspecto:

∅  ansible [pattern]-m [module] -a "[module-options]" 
∅ REINICIAR SERVIDORES

El módulo por defecto para la utilidad de línea de comandos de Ansible es el módulo ansible.builtin.command.

Una tarea ad hoc puede llamar al módulo de comandos y reiniciar todos los servidores web del grupo [laboratorio], de 10 en 10.

Antes de que Ansible pueda hacer esto, se debe crear un inventario con todos los servidores listados en un grupo llamado, por ejemplo, [laboratorio], y se debe copiar la ssh-key del nodo de control en el fichero authorized_keys de los nodos administrados.

• INVENTARIO:

El inventario de este post contiene un nodo administrado con dirección IP 192.168.1.115 y versión Python39 instalada

ansible_python_interpreter indica en qué ruta se encuentra Python en el nodo administrado.

ansible_user especifica el usuario que ejecutará las tareas en el nodo administrado.
∅  cat /tmp/inventario

[laboratorio]
192.168.1.115 ansible_python_interpreter=/usr/bin/python3 ansible_user=root 


• SSH-KEYS:

Es necesario copiar las ssh-keys del nodo de gestión a los nodos administrados para que Ansible pueda comunicarse con ellos:

∅  ssh-copy-id root@192.168.1.115

/usr/bin/sshd-copy-id: INFO: 1 key(s) remain to be installed 
root@192.168.1.115`s password:
Number of key(s) added: 1


Para reiniciar todos los servidores del grupo [laboratorio]:
∅  ansible laboratorio -i /tmp/inventario -a "/sbin/reboot" 


Por defecto, Ansible realiza 5 procesos simultáneos {forks}.

Si el inventario contiene, por ejemplo, 10 hosts, para reiniciar los servidores del grupo [laboratorio] de 10 en 10:
∅  ansible laboratorio -i /tmp/inventario -a "/sbin/reboot" -f 10 



Si no se ha definido un usuario en la variable ansible_user del inventario, se deberá especificar el usuario usando la opción -u en cada tarea:
∅  ansible laboratorio -i /tmp/inventario -a "whoami" -u usuario 


Tareas como el reinicio de servidores pueden requerir privilegios elevados como sudo, por parte de un usuario.

La opción become le dice a Ansible que la tarea requiere una escalada de privilegios para llevarse a cabo, pero no pedirá la password del usuario privilegiado:

∅  ansible laboratorio -i /tmp/inventario -a "/sbin/reboot" -f 10 -u usuario --become


Si la tarea requiere introducir la password del usuario privilegiado, será necesaria la opción --ask-become-pass o -K:

∅  ansible laboratorio -i /tmp/inventario -a "/sbin/reboot" -f 10 -u usuario --become -K

NOTA: Si el usuario especificado no tiene privilegios sudo en el nodo administrado, la tarea no se ejecutará aún especificando la opción --become.
∅ ADMINISTRAR FICHEROS

• TRANSFERIR ARCHIVOS:

Una tarea ad hoc puede aprovechar la potencia de Ansible y SCP para transferir múltiples archivos a varias máquinas en paralelo. Para transferir un archivo directamente a todos los servidores del grupo [laboratorio] con el módulo ansible.builtin.copy:

∅  ansible laboratorio -i /tmp/inventario -m copy -a "src=/etc/hosts dest=/tmp" -u usuario

El módulo ansible.builtin.file permite modificar los permisos de archivos:

∅  ansible laboratorio -i /tmp/inventario -m file -a "dest=/tmp/hosts mode=600" -u usuario --become -K

• ELIMINAR ARCHIVOS:

NOTA: Para eliminar el archivo /tmp/hosts en este punto, no será necesario especificar la opción --become porque en el paso anterior le hemos dado permisos 777. Cualquier usuario puede manipular el fichero sin necesidad de privilegios:
∅  ansible laboratorio -i /tmp/inventario -m file -a "dest=/tmp/hosts state=absent" -u usuario

∅ ADMINISTRAR SERVICIOS

El módulo service permite administrar servicios en el nodo adminsitrado.

• INICIAR UN SERVICIO:

∅  ansible laboratorio -i /tmp/inventario -m service -a "name=nginx state=started" --become -K 
SUDO password:

∅ FACTS

Los facts representan variables descubiertas sobre un sistema. Los facts permiten implementar la ejecución condicional de tareas y obtener información ad hoc sobre los sistemas.

El módulo ansible.builtin.setup es el encargado de trabajar con facts:

∅  ansible laboratorio -i /tmp/inventario -m setup 
centos8.ansible.com | SUCCESS => {
  "ansible_facts": {
    "ansible_all_ipv4_addresses": [
         "192.168.1.54"
  ],

},

  • 192.168.1.115: IP del nodo administrado
  • -i: Inventario
  • -m: Módulo
  • -m: Forks
  • ansible_python_interpreter: Variable del inventario que define la ruta del intérprete de Python en el nodo administrado

  • -K: Opción abreviada de --ask-become-pass
